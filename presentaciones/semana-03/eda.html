Preprocesamiento: limpieza de datos

La fase que consume 60-70% del tiempo del proyecto

Principio fundamental

El preprocesamiento es como "editar un documento antes de enviarlo" — identifica y corrige errores para que el análisis produzca resultados confiables. Ningún algoritmo puede compensar datos sucios.

Manejo de valores faltantes

Método Qué hace Cuándo usar
Eliminación Remueve filas/columnas con datos faltantes Menos de 5% faltante, dispersión aleatoria
Imputación por media/mediana Llena vacíos con valores promedio Datasets simples, datos numéricos
Imputación por moda Llena con valor más común Datos categóricos
K-Nearest Neighbors Usa registros similares para estimar Cuando existen patrones en datos

Manejo de outliers

Pregunta crítica: ¿es un error o un evento genuino raro?

Ejemplo 1: En detección de fraude, los outliers pueden ser exactamente lo que buscamos

Ejemplo 2: En pronóstico de ventas, pueden distorsionar predicciones

Validación de datos

  • Tipos de datos: ¿"edad" almacena números o texto?
  • Rangos válidos: ¿todos los precios son positivos?
  • Validación cruzada: ¿"fecha fin" viene después de "fecha inicio"?

Preprocesamiento: normalización

Poniendo todo en la misma escala

Analogía simple

Imagina comparar calificaciones de dos clases — una en escala de 100 puntos, otra en escala de 1000 puntos. La normalización pone todo en la misma escala para que las comparaciones sean justas.

Por qué importa para ML

Sin normalización, características con números más grandes (como salario anual: $50,000-$150,000) dominarán características más pequeñas (como años de experiencia: 1-30). Muchos algoritmos asumen que todas las características contribuyen igualmente.

Min-Max Scaling

Convierte todos los valores a escala 0-1:

  • Edad 20 (mínima) → 0.0
  • Edad 40 (media) → 0.5
  • Edad 60 (máxima) → 1.0

Mejor para: Algoritmos sensibles a escala (redes neuronales, k-NN, SVM)

Z-Score (Standardization)

Centra datos alrededor de cero midiendo "cuántas desviaciones estándar del promedio":

  • Z-score de 0 = exactamente promedio
  • Z-score de +1 = una desviación arriba
  • Z-score de -2 = dos desviaciones abajo

Mejor para: Datos con distribución normal, algoritmos basados en distancia

Regla práctica: Normalizar siempre que las características tengan unidades o escalas diferentes. No normalizar datos que ya están en la misma escala o cuando el modelo es invariante a escala (árboles de decisión, Random Forest).

Preprocesamiento: encoding

Convirtiendo categorías a números

El problema

Los algoritmos ML trabajan con números, pero los datos de negocio contienen categorías como "Rojo/Azul/Verde" o "Electrónicos/Ropa/Alimentos". Necesitamos convertir categorías a representaciones numéricas.

One-Hot Encoding

Recomendado para la mayoría de casos de negocio

Crea una columna sí/no separada para cada categoría:

Original Es_Norte Es_Sur Es_Este Es_Oeste
Norte 1 0 0 0
Sur 0 1 0 0

Mejor para: Datos nominales (categorías sin orden natural) — productos, colores, países

Label Encoding

Asigna un número a cada categoría:

  • Apple = 0
  • Banana = 1
  • Orange = 2

Mejor para: Datos ordinales (categorías con orden natural)

  • Satisfacción: Pobre=1, Regular=2, Bueno=3, Excelente=4
  • Tallas: S=1, M=2, L=3, XL=4

⚠️ Advertencia: No usar Label Encoding para datos nominales — el modelo podría interpretar incorrectamente que "Orange (2)" es "mayor que" Apple (0) cuando no hay relación ordinal.

Análisis exploratorio de datos (EDA)

El trabajo detectivesco antes de construir modelos

Qué es EDA

El Análisis Exploratorio de Datos es el paso crítico donde analistas investigan datasets para entender sus características principales, descubrir patrones, detectar anomalías y probar suposiciones — principalmente usando métodos visuales. Es el "trabajo detectivesco" antes de construir cualquier modelo.

80%
del tiempo los científicos de datos lo pasan en limpieza y exploración
1970s
desarrollado por el matemático John Tukey

Por qué es crítico hacer EDA

Saltar EDA frecuentemente lleva a:

  • Construir modelos sobre distribuciones sesgadas — sin entender los datos reales
  • Perder outliers críticos que señalan fraude o fallas de equipos
  • Desperdiciar semanas en características que no correlacionan con resultados
  • No detectar problemas de calidad hasta que es demasiado tarde

EDA responde la pregunta fundamental: "¿Qué nos están diciendo estos datos antes de hacer suposiciones?"

Técnicas principales de EDA

Del análisis univariado al multivariado

Análisis univariado

Una variable a la vez

Ejemplo: Antes de analizar qué clientes abandonan, entender el perfil básico de la base de clientes — distribución de edad, patrones de gasto, dispersión geográfica

Análisis bivariado

Relaciones entre dos variables

Ejemplo: Entender la relación entre gasto en marketing y ventas, o antigüedad del cliente y frecuencia de compra

Análisis multivariado

Múltiples variables simultáneamente

Ejemplo: Cómo edad, ingreso y frecuencia de compra juntos predicen valor de vida del cliente

Análisis de correlación

Medir la fuerza y dirección de relaciones entre variables (de -1 a +1).

⚠️ Correlación NO equivale a causalidad — las ventas de helado correlacionan con incidentes de ahogamiento, pero ambos son causados por el clima veraniego.

Insights de negocio descubiertos mediante EDA

  • Walmart: las ventas de Pop-Tarts de fresa aumentaban 7x antes de huracanes
  • Netflix: alta demanda de dramas políticos + afinidad por Kevin Spacey → "House of Cards"
  • Cliente retail: 15% de "clientes de alto valor" eran abusadores de devoluciones (ahorro de $2.7M anuales)
  • Manufactura: anomalías de vibración precedían 92% de fallas por 36 horas ($4.2M ahorro anual)

Comunicando insights de manera efectiva

Por qué la visualización es esencial

La visualización cumple roles críticos en tres etapas del ciclo ML: comprensión de datos (patrones invisibles en datos crudos se vuelven evidentes), desarrollo de modelos (identificar overfitting/underfitting), y comunicación de resultados (convencer stakeholders).

En comprensión de datos

  • Histogramas revelan distribuciones
  • Box plots identifican outliers
  • Scatter plots muestran relaciones
  • Heatmaps revelan correlaciones

En comunicación de resultados

  • Matrices de confusión muestran errores
  • Curvas ROC evalúan clasificadores
  • Feature importance guía decisiones
  • Dashboards interactivos para stakeholders

La visualización efectiva transforma datos complejos en insights accionables que impulsan decisiones de negocio.

En 2025, Google lanzó el Data Science Agent impulsado por Gemini que puede automatizar EDA completo con prompts en lenguaje natural. Le describes lo que quieres hacer, y Gemini genera el código Python, ejecuta el análisis y crea visualizaciones.

Qué puede hacer Gemini

  • Explicar código existente: entender qué hace cada línea
  • Sugerir mejoras: optimizar código para mejor rendimiento
  • Depurar errores: identificar y corregir bugs

Ejemplo de prompts

Herramientas emergentes: Polars y DuckDB

La próxima generación de procesamiento de datos

Polars: Pandas acelerado

Qué es: Biblioteca de procesamiento de datos escrita en Rust, compatible con sintaxis similar a Pandas

Ventaja principal: 7.7x más rápido que Pandas para CSVs grandes

Características:

  • Evaluación lazy (ejecuta solo cuando necesario)
  • Paralelización automática
  • Uso eficiente de memoria

DuckDB: SQL analítico

Qué es: Base de datos analítica SQL-nativa embebida

Ventaja principal: 2-6x más rápido que Spark para analítica

Características:

  • SQL estándar completo
  • Sin servidor (embedded)
  • Queries directos sobre Parquet, CSV

Ejemplo comparativo de velocidad

# Pandas (tradicional) import pandas as pd df = pd.read_csv('large_file.csv') result = df.groupby('category')['sales'].sum() # Polars (7.7x más rápido) import polars as pl df = pl.read_csv('large_file.csv') result = df.groupby('category').agg(pl.col('sales').sum()) # DuckDB (SQL directo sobre archivo) import duckdb result = duckdb.sql(""" SELECT category, SUM(sales) FROM 'large_file.csv' GROUP BY category """).df()

Cuándo usar qué: Pandas para datasets pequeños/medianos y prototipado. Polars para datasets grandes con operaciones complejas. DuckDB para analítica SQL sobre archivos grandes sin cargar en memoria.

Mejores prácticas de EDA para negocios

Cómo hacer EDA de manera efectiva y eficiente

1. Comience con preguntas de negocio claras

No explore datos sin dirección. Defina qué quiere descubrir:

  • ¿Qué factores influyen más en la retención de clientes?
  • ¿Existen segmentos de clientes con comportamientos distintos?
  • ¿Qué productos se compran frecuentemente juntos?

2. Siga una estructura sistemática

  • Entender el contexto de los datos
  • Inspeccionar dimensiones y tipos
  • Identificar valores faltantes y outliers
  • Analizar distribuciones univariadas
  • Explorar relaciones bivariadas
  • Buscar patrones temporales
  • Documentar hallazgos

3. Visualice antes de modelar

  • Una imagen vale más que mil estadísticas
  • Patrones ocultos se revelan visualmente
  • Valide suposiciones con gráficos
  • Comunique hallazgos efectivamente
  • Use múltiples tipos de gráficos

4. Documente todo el proceso

Cree notebooks reproducibles que otros puedan seguir:

  • Explique el propósito de cada análisis
  • Documente decisiones de limpieza
  • Anote hallazgos importantes
  • Incluya interpretaciones de negocio
  • Versione sus notebooks con Git

Regla de oro: El EDA no es lineal — itere entre exploración, limpieza y validación hasta entender completamente sus datos.

Errores comunes en EDA y cómo evitarlos

Aprenda de los errores más frecuentes

❌ Error 1: Confiar ciegamente en la estadística descriptiva

El "Cuarteto de Anscombe" demuestra que datasets con estadísticas idénticas pueden tener distribuciones completamente diferentes.

Solución: Siempre visualice sus datos. Los gráficos revelan lo que las estadísticas ocultan.

❌ Error 2: Ignorar el contexto de negocio

Un outlier de $1M en ventas puede ser un error... o su mejor cliente.

Solución: Consulte con expertos de dominio antes de eliminar datos "anómalos".

❌ Error 3: No verificar suposiciones

Asumir que los datos son representativos sin verificar sesgos de muestreo.

Solución: Pregunte: ¿Cómo se recolectaron estos datos? ¿Qué población representan?

❌ Error 4: Correlación implica causalidad

Las ventas de helado correlacionan con ahogamientos — ambos aumentan en verano.

Solución: Use correlación para hipótesis, no para conclusiones. Valide causalidad con experimentos.

❌ Error 5: Data leakage en exploración

Usar datos de test durante EDA contamina la evaluación del modelo.

Solución: Separe train/test ANTES de EDA. Explore solo el conjunto de entrenamiento.

La mayoría de errores en ML se originan en EDA deficiente, no en selección de algoritmos.
